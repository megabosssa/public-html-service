SOLID Principles

The foundational principles of object-oriented design

Single Responsibility Principle (SRP)

A class should have one, and only one, reason to change. Meaning that a class should have only one job.

Java - Example 
// ‚ùå Bad: Handles both business logic and persistence
class Employee {
    public void calculatePay() { /*...*/ }
    public void saveToDatabase() { /*...*/ }
}

// ‚úÖ Good: Responsibilities are separated
class Employee {
    public void calculatePay() { /*...*/ }
}

class EmployeeRepository {
    public void save(Employee employee) { /*...*/ }
}

Open/Closed Principle (OCP)

Objects or entities should be open for extension but closed for modification. You should be able to add new functionality without changing existing code.

Java - Example 
// ‚úÖ Good: Strategy pattern to allow extending without modification
interface DiscountStrategy {
    double apply(double amount);
}

class VIPDiscount implements DiscountStrategy {
    @Override
    public double apply(double amount) { return amount * 0.8; }
}

// The checkout service doesn't need to change when a new discount type is added
class CheckoutService {
    public double calculateTotal(double amount, DiscountStrategy discount) {
        return discount.apply(amount);
    }
}

Liskov Substitution (LSP)

Subclasses should be replaceable for their base classes without altering the correctness of the program.

Java - Example 
// ‚ùå Bad: Throwing exception violates LSP
class Bird {
    public void fly() { /*...*/ }
}

class Penguin extends Bird {
    @Override
    public void fly() { 
        throw new UnsupportedOperationException("Cannot fly"); 
    }
}

// ‚úÖ Good: Refactor hierarchy to respect behavior
interface FlyingBird {
    void fly();
}

class Eagle implements FlyingBird {
    public void fly() { /*...*/ }
}

Interface Segregation (ISP)

A client should never be forced to implement an interface that it doesn't use. Break large interfaces into smaller, more specific ones.

Java - Example 
// ‚ùå Bad: One fat interface
interface Worker {
    void work();
    void eat();
}

class Robot implements Worker {
    public void work() { /*...*/ }
    public void eat() { /* Not applicable */ }
}

// ‚úÖ Good: Segregated interfaces
interface Workable { void work(); }
interface Eatable { void eat(); }

class Human implements Workable, Eatable { /*...*/ }
class Robot implements Workable { /*...*/ }

Dependency Inversion (DIP)

High-level modules must not depend on low-level modules; both should depend on abstractions. Crucial for Microservices and Spring Boot.

Java Spring - Example 
// ‚ùå Bad: Tightly coupled to a specific implementation
class OrderService {
    private MySQLOrderRepository repository = new MySQLOrderRepository();
}

// ‚úÖ Good: Depends on abstraction, injected via constructor
@Service
class OrderService {
    private final OrderRepository repository;

    // Spring will inject the implementation here
    public OrderService(OrderRepository repository) {
        this.repository = repository;
    }
}

Clean Code

Write code that is easy to read, understand, and maintain

DRY vs WET Code

DRY (Don't Repeat Yourself) is the principle of reducing repetition of software patterns. WET (Write Everything Twice or We Enjoy Typing) is the opposite.

Java - Example 
// ‚ùå Bad (WET): Repeating validation logic
public void createUser(UserDto dto) {
    if (dto.getEmail() == null || !dto.getEmail().contains("@")) { throw new Exception(); }
    /*...*/
}
public void updateUser(UserDto dto) {
    if (dto.getEmail() == null || !dto.getEmail().contains("@")) { throw new Exception(); }
    /*...*/
}

// ‚úÖ Good (DRY): Extract to a method or use Spring Boot Validation (@Valid)
private void validateEmail(String email) {
    if (email == null || !email.contains("@")) { throw new ValidationException(); }
}
Rule of Three: Write it once. Write it twice, you might feel a bit of pain. When you write it a third time, it's time to extract it into a reusable function or module.

KISS (Keep It Simple, Stupid)

Most systems work best if they are kept simple rather than made complicated; therefore, simplicity should be a key goal in design, and unnecessary complexity should be avoided.

Java - Example 
// ‚ùå Bad (Over-engineered for a simple task)
String names = users.stream()
    .filter(u -> Optional.ofNullable(u.getAge()).orElse(0) > 18)
    .map(User::getName)
    .collect(Collectors.joining(", "));

// ‚úÖ Good (Simple and readable)
List<String> adultNames = new ArrayList<>();
for (User user : users) {
    if (user.getAge() != null && user.getAge() > 18) {
        adultNames.add(user.getName());
    }
}
String names = String.join(", ", adultNames);

YAGNI (You Aren't Gonna Need It)

Always implement things when you actually need them, never when you just foresee that you need them. Unused code is a liability, not an asset.

Java - Example 
// ‚ùå Bad: Adding generic type parameters "just in case" we need to support other databases in the future
public interface GenericRepository<T, ID extends Serializable, DB_TYPE extends Database> {
    /*...*/
}

// ‚úÖ Good: Start concrete, extract interface only when the second implementation arrives
public interface UserRepository extends JpaRepository<User, Long> {
    /*...*/
}

Meaningful Naming

  • Use intention-revealing names: elapsedTimeInDays instead of d.
  • Avoid encodings and prefixes like m_ or I_.
  • Class names should be nouns (e.g., Customer, Account).
  • Method names should be verbs (e.g., postPayment, deletePage).
Java - Example 
// ‚ùå Bad
class DataProc {
    public void run() { /*...*/ }
    public List<Object> get() { /*...*/ }
}

// ‚úÖ Good
class EndOfDayReportGenerator {
    public void generateReport() { /*...*/ }
    public List<TransactionDto> getPendingTransactions() { /*...*/ }
}

Design Patterns

Reusable solutions to common software design problems

1. Strategy Pattern (Behavioral)

Defines a family of algorithms, encapsulates each one, and makes them interchangeable.

Java - Strategy Use Case 
// Useful for substituting different payment methods at runtime
interface PaymentStrategy {
    void pay(int amount);
}

class CreditCardPayment implements PaymentStrategy {
    public void pay(int amount) { System.out.println("Paid with CC: " + amount); }
}

class PaypalPayment implements PaymentStrategy {
    public void pay(int amount) { System.out.println("Paid with PayPal: " + amount); }
}

2. Observer Pattern (Behavioral)

Define a one-to-many dependency between objects so that when one object changes state, all its dependents are notified.

When to use: Event handling systems, UI rendering based on state changes, publish/subscribe mechanisms (like Kafka or RabbitMQ event listeners).

3. Builder Pattern (Creational)

Separate the construction of a complex object from its representation, allowing the same construction process to create various representations.

Java - Builder Use Case 
// Very common in Spring Boot and Lombok!
User user = User.builder()
    .id(1L)
    .firstName("John")
    .lastName("Doe")
    .email("john@example.com")
    .build();

Code Smells

Indicators that something might be wrong with your design

God Class

A class that controls way too many other classes and handles too many responsibilities. Violates SRP. Solution: Extract classes based on behavior.

Long Method

A method containing too many lines of code. Hard to read and test. Solution: Extract Method to break it down into smaller named functions.

Feature Envy

A method accesses the data of another object more than its own data. Solution: Move the method to the class where the data resides.

Magic Numbers

Unexplained numbers directly hard-coded in logic. Solution: Replace with named constants or enumerations.
